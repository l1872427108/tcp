# Nagle算法和延迟确认

如果发送端一直不停的发送，并且每次发送的包只有1个字节，那么发发送1千个字节就需要发送1千次。

频繁发送不仅带来传输的时延消耗，发送和确认也是需要时间的，频繁的发送会带来巨大的时延。

避免小包的频繁发送，就是 Nagle 算法要做的事情。

  1. 当第一次发送数据时不用等待，就算 1 个字节也要立即发送。
  2. 后面发送要满足以下条件：
    • 数据包大小达到最大段大小。 MSS
    • 之前所有包的 ACK 都已经收到。

# 延迟确认

如果发送了一个包，很短的时间又收到一个包，那么是一个个的回复，还是做一次性的回复呢？

延迟确认，就是将两个包合并 ACK 回复发送端，但需要稍稍的延迟。TCP 要求这个延迟的时延必须小于 500ms，一般操作系统都不会超过 200ms。

有些情况下，收到了就必须马上回复：

  • 接收到了大于一个 frame 的报文，且需要调整窗口大小.
  • TCP 处于 quickack 模式。
  • 发现了乱序包。

如果使用延迟确认，再加上后者的延迟，将会造成更大的延迟。
